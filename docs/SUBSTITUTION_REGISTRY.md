# Custom-Component Substitution Registry (M7)

When a customer's SSIS estate contains 3rd-party (Cozyroc, KingswaySoft) or
in-house Script / pipeline components, the agent's default behaviour is one
of:

* **Unknown component type** → emits a placeholder `DerivedColumn` and a
  warning, *or*
* **Unsupported component type** → emits a placeholder `Wait` and a warning.

Either way the customer has to hand-edit ADF before deploying. The
substitution registry lets them write the mapping down once, in JSON, and
have every conversion apply it deterministically.

---

## Schema

```json
{
  "version": "1",
  "data_flow_components": {
    "<SSIS component_type string>": {
      "adf_type": "<ADF MDF transformation type, e.g. Lookup, Filter, DerivedColumn>",
      "notes": "Free-text reviewer note, surfaced in pipeline description.",
      "type_properties": { "any": "ADF type-properties block to inject" }
    }
  },
  "control_flow_tasks": {
    "<SSIS task creation-name (GUID or string)>": {
      "adf_activity_type": "<ADF activity type, e.g. WebActivity, AzureFunctionActivity>",
      "notes": "Free-text reviewer note.",
      "type_properties": { "url": "@pipeline().parameters.endpoint" }
    }
  }
}
```

Either map can be empty / omitted.

## Worked example — Cozyroc Salesforce source replaced by ADF Copy

```json
{
  "version": "1",
  "data_flow_components": {
    "Cozyroc.SSISPlus.SalesforceSource": {
      "adf_type": "Source",
      "notes": "Replaced with ADF Salesforce connector — wire DS_Salesforce_<Object> by hand.",
      "type_properties": {
        "schemaLinkedService": { "referenceName": "LS_Salesforce", "type": "LinkedServiceReference" }
      }
    }
  }
}
```

Pass it in via the MCP tool:

```json
{
  "package_path": "ssis/MyPackage.dtsx",
  "output_dir": "out/MyPackage",
  "substitution_registry_path": "config/substitution_registry.json"
}
```

Or via the headless CLI (M3) by editing the generated artifacts after — the
flag is currently exposed only through the MCP `convert_ssis_package` tool;
file an issue if you need it on the CLI too.

## What the registry does *not* do

* It does not magically wire the new ADF transformation's columns. The
  customer / reviewer is responsible for completing the substitution.
* It does not validate that `adf_type` / `adf_activity_type` is real — typos
  will surface only when ADF rejects the JSON at deploy time.
* It does not silence warnings about adjacent unsupported components — only
  the substituted one.

## Why JSON, not Python plugins?

A JSON registry means:

* No `pip install` step on every developer's machine.
* The file lives next to the .dtsx packages it applies to, in version
  control, reviewed in PRs.
* It can be generated by other tooling (Excel export, scan of an existing
  `Cozyroc.dll`, etc.) without depending on the agent's Python ABI.
